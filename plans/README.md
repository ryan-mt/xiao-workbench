## Critical path

```text
M0 -> M1 -> M2 -> M3 -> M4 -> M5
```

This path ships Xiao's first defensible differentiation:

```text
Goal -> isolated durable run -> routine -> deterministic verification -> evidence
```

After M3, M6 may proceed separately. M7 requires the verification lifecycle from
M5. M8 must wait for isolation, concurrency and verification. M9 must not delay
the Codex-first critical path.

## Planning envelope

Assuming one experienced maintainer:

- Critical path through M5: approximately 10–16 maintainer-weeks.
- Full roadmap when executed serially: approximately 22–32 maintainer-weeks.
- Estimates include tests/docs but not code-signing or store certification.
- Expect ±50% variance around Codex protocol and WSL unknowns.

## Mandatory rules

- Create feature branches from `dev`; target PRs to `dev`.
- Before each milestone, create a self-contained child plan in `plans/` using
  the next monotonic number.
- Do not implement multiple milestones in one PR.
- Run all verification commands from Plan 001 before each milestone exits.
- Never bypass approvals, workspace scoping or worktree ownership checks.
- Do not add analytics, cloud storage or remote execution as a shortcut.

## Findings considered and rejected

- Full mobile/cloud/SSH/Tailscale stack: not worth delaying Xiao's local moat.
- Direct catalog of dozens of LLM providers: use one runtime adapter instead.
- Full IDE/LSP/editor: integrate with a preferred editor; keep Xiao a workbench.
- Plugin marketplace: defer until repeated extension demand exists.
- Arbitrary cron/external triggers in Routines v1: defer until recurrence is restart-safe.
- Automatic Task Arena merge: reject initially because verification cannot guarantee merge correctness.
- Windows Service execution while Xiao is fully exited: defer; tray residency first.
- Additional Xiao Break games: defer until autonomy/recovery is stable.
